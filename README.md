# OPENAPI CLIENT (GPRO)

## Introducción

Instrucciones básicas para la generación de un cliente de una API a partir del yaml con openapi-generator-cli

## Generación de código

### Cliente
Nuestro objetivo es generar un cliente Java (17) con nuestra paquetería java de siempre (org.jlobato.gpro). Para ello, elcomando final utilizando el fichero openapi-generator-cli.jar sería este:

```sh
java -jar ${path-openapi}/openapi-generator-cli.jar generate \
  -i gpro-public-api-fixed.yml \
  -g java \
  -o gpro-client \
  --library webclient \
  --additional-properties=useJakartaEe=true,dateLibrary=java8,serializationLibrary=jackson,hideGenerationTimestamp=true \
  --api-package org.jlobato.gpro.api \
  --model-package org.jlobato.gpro.model \
  --invoker-package org.jlobato.gpro.client \
  --ignore-file-override=.openapi-generator-ignore \
  --additional-properties=groupId=org.jlobato.gpro \
  --additional-properties=artifactId=gpro-client \
  --additional-properties=artifactVersion=1.0.0.RELEASE \
  --additional-properties=artifactDescription="Java client por GPRO API generated by OpenAPI"
```
Para facilidad, en una sola línea:

```sh
java -jar ${path-openapi}/openapi-generator-cli.jar generate -i gpro-public-api-fixed.yml -g java -o gpro-client --library webclient --additional-properties=useJakartaEe=true,dateLibrary=java8,serializationLibrary=jackson,hideGenerationTimestamp=true --api-package org.jlobato.gpro.api --model-package org.jlobato.gpro.model --invoker-package org.jlobato.gpro.client --ignore-file-override=.openapi-generator-ignore --additional-properties=groupId=org.jlobato.gpro --additional-properties=artifactId=gpro-client --additional-properties=artifactVersion=1.0.0.RELEASE --additional-properties=artifactDescription="Java client por GPRO API generated by OpenAPI"
```
En las siguientes secciones explicamos cómo hemos llegado a este comando.

### Proxy

El objetivo es generar un API Proxy basado en Spring Boot 3 para hacer llamadas a la API. El comando final al que hemos llegado es el siguiente:

```sh
java -jar ${path-openapi}/openapi-generator-cli.jar generate \
  -i gpro-public-api-fixed.yml \
  -g spring \
  -o gpro-proxy \
  --library spring-boot \
  --additional-properties=useSpringBoot3=true,interfaceOnly=true,delegatePattern=true,useJakartaEe=true,useTags=true \
  --additional-properties=openApiNullable=false,documentationProvider=none \
  --additional-properties=hideGenerationTimestamp=true,configPackage=org.jlobato.gpro.proxy.config \
  --api-package org.jlobato.gpro.proxy.api \
  --model-package org.jlobato.gpro.proxy.model \
  --ignore-file-override=.openapi-generator-ignore \
  --additional-properties=groupId=org.jlobato.gpro \
  --additional-properties=artifactId=gpro-proxy \
  --additional-properties=artifactVersion=1.0.0.RELEASE \
  --additional-properties=artifactDescription="Spring Boot 2 API Proxy por GPRO API generated by OpenAPI" \
  --global-property apis,models
```
Para facilidad, en una sola línea:

```sh
java -jar ${path-openapi}/openapi-generator-cli.jar generate -i gpro-public-api-fixed.yml -g spring -o gpro-proxy --library spring-boot --additional-properties=useSpringBoot3=true,interfaceOnly=true,delegatePattern=true,useJakartaEe=true,useTags=true,openApiNullable=false,documentationProvider=none,hideGenerationTimestamp=true,configPackage=org.jlobato.gpro.proxy.config --api-package org.jlobato.gpro.proxy.api --model-package org.jlobato.gpro.proxy.model --ignore-file-override=.openapi-generator-ignore --additional-properties=groupId=org.jlobato.gpro --additional-properties=artifactId=gpro-proxy --additional-properties=artifactVersion=1.0.0.RELEASE --additional-properties=artifactDescription="Spring Boot 2 API Proxy por GPRO API generated by OpenAPI" --global-property apis,models
```

### Comando base cliente

El comando base sería:

```sh
openapi-generator-cli generate -i gpro-public-api.yml -g java -o gpro-api-client --additional-properties=library=webclient,dateLibrary=java8,useJakartaEe=true
```

siendo gpro-public-api.yml el yaml de partida

### Spring Boot 3+ y Java 17

Si usamos Spring Boot 3 / Java 17, lo recomendado es: 

```sh
library=webclient
```

```sh
useJakartaEe=true
```

### Uso de java.time

Para usar java.time lo suyo es

```sh
dateLibrary=java8
```

### Limpieza de código

Para obtener un código más limpio, añadir también:

```sh
--additional-properties=hideGenerationTimestamp=true,serializationLibrary=jackson
```

Con el primero no se mete la fecha de generación de los archivos lo que hace que a la hora de hacer los diffs sea más sencillo.

### Resumen

--additional-properties se usa para: 

| Propiedad                     | Motivo                          |
|------------------------------|----------------------------------|
| useJakartaEe=true            | Compatible con Spring Boot 3+    |
| dateLibrary=java8            | Usa java.time                    |
| serializationLibrary=jackson | Estándar en Spring               |
| hideGenerationTimestamp=true | Evita ruido en diffs             |

### Paquetería java

Para generar el cliente con los paquetes java que nos interesan lo suyo es utilizar los siguientes parámetros del cliente

```sh
--api-package org.jlobato.gpro.api \
--model-package org.jlobato.gpro.model \
--invoker-package org.jlobato.gpro.client \
```

### Personalización del artefacto (pom.xml)

Para personalizar el pom.xml a nuestro gusto, añadimos los siguientes parámetros en la generación (groupId, artifactId, versión, etc.)

```sh
  --additional-properties=groupId=org.jlobato.gpro \
  --additional-properties=artifactId=gpro-api \
  --additional-properties=artifactVersion=1.0.0.RELEASE \
  --additional-properties=artifactDescription="Java client por GPRO API generated by OpenAPI"
```

## Generación del cliente en el build de maven

La generación del cliente se puede automatizar con maven usando el plugin correspondiente:

```xml
<plugin>
    <groupId>org.openapitools</groupId>
    <artifactId>openapi-generator-maven-plugin</artifactId>
    <version>7.5.0</version>
    <executions>
        <execution>
            <goals>
                <goal>generate</goal>
            </goals>
            <configuration>
                <inputSpec>${project.basedir}/src/main/resources/gpro-public-api.yml</inputSpec>
                <generatorName>java</generatorName>
                <library>webclient</library>
                <output>${project.build.directory}/generated-sources/openapi</output>
                <configOptions>
                    <dateLibrary>java8</dateLibrary>
                    <useJakartaEe>true</useJakartaEe>
                </configOptions>
            </configuration>
        </execution>
    </executions>
</plugin>
```

Y luego:

```xml
<build>
    <plugins>
        ...
    </plugins>
</build>

<dependencies>
    ...
</dependencies>
```

Y en el build:
```xml
<build>
    <sourceDirectory>${project.build.directory}/generated-sources/openapi/src/main/java</sourceDirectory>
</build>
```

## Utilización del wrapper de maven

Para dejar el proyecto generado preparado para utilizar maven wrapper basta con hacer lo siguiente:

```sh
cd gpro-api-client
mvn -N wrapper:wrapper
```

Este error se debe a que la propiedad `offersFrom` de la entidad `DriverProfileResponse` no está correcta


```yml
offersFrom:

```

La indentación de type es importante. Si no se indenta dará un error de propiedad type duplicada.

 Se debe a que está generando un parámetro con el nombre reservado false. Esto es por este trozo de código en el yaml:

 ```yml

 ```

 ```yml
    name: offers
    in: query
    description: >
 ```

 Y se corrige.

 ### Error en método duplicado en DriverProfileResponse, TDProfileResponse y GetDriverProfile200Response

Esto se debe a que hay colisiones con los nombres de las propiedades y los métodos:

```yml
offerCost:
  type: integer
semiOfferCost:
  type: integer
privateOfferCost:
  type: integer
setOfferCost:
  type: integer
```

El problema está en offerCost y setOfferCost ambas colisionan. Basta con cambiar setOfferCost por offerCostToSet (por ejemplo):

```yml
offerCost:
  type: integer
semiOfferCost:
  type: integer
privateOfferCost:
  type: integer
offerCostToSet:
  type: integer
```

Con estos dos cambios ya compila correctamente el código generado por openapi generator
